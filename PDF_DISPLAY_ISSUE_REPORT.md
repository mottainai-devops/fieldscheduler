# Zoho Books Customer Statement PDF Display Issue - Technical Report

**Date**: November 1, 2025  
**Status**: BLOCKED - PDF pages render blank despite successful API calls and PDF generation  
**Priority**: HIGH - Core feature not working

---

## Executive Summary

The Field Worker Scheduler application has successfully integrated with Zoho Books API to fetch customer statements and generate PDFs. However, **the PDF content is not displaying in the browser** - pages show as blank despite:
- ✅ Successful Zoho API calls returning valid statement data
- ✅ PDF generation completing without errors
- ✅ Binary PDF being served with correct HTTP headers
- ✅ Browser PDF viewer loading the PDF (page count shows 1/3)

---

## Problem Statement

**What's Not Working:**
- Customer statement PDFs display as blank pages in the browser iframe
- PDF viewer shows page navigation (1/3 pages) but content is invisible
- The issue occurs consistently across multiple customers with valid Zoho data

**What IS Working:**
- Zoho Books API authentication and token refresh
- Statement data fetching (JSON format with invoices, contact info, etc.)
- PDF generation using Puppeteer (no errors in logs)
- PDF binary endpoint serving files with correct Content-Type headers
- Frontend iframe loading the PDF (no 404 or CORS errors)

---

## Technical Architecture

### Backend Stack
- **Framework**: Express.js with Node.js
- **PDF Generation**: Puppeteer (headless Chrome) + pdf-lib
- **API Integration**: Zoho Books REST API
- **Database**: MySQL with Drizzle ORM

### Frontend Stack
- **Framework**: React 19 with TypeScript
- **PDF Display**: Native browser PDF viewer via `<iframe>`
- **API Client**: tRPC for type-safe API calls

### Data Flow
```
1. Frontend: User clicks "Zoho Statement" tab
2. Frontend: Calls tRPC endpoint with customer's zohoContactId
3. Backend: Fetches statement data from Zoho Books API (JSON)
4. Backend: Converts JSON to HTML using generateStatementHTML()
5. Backend: Uses Puppeteer to render HTML → PDF
6. Backend: Returns { pdfBase64, zohoContactId, total, balance, invoices }
7. Frontend: Renders iframe with src="/api/statements/{zohoContactId}.pdf"
8. Backend: PDF endpoint converts base64 to binary and serves with proper headers
9. Browser: Loads PDF in viewer (shows 1/3 pages but content is blank)
```

**Status**: Suspected issue

**Possible Issue**: Puppeteer might not be rendering the HTML content correctly:
- CSS might not be applying properly
- Text color might be white on white background
- HTML structure might have rendering issues in headless browser
- Font embedding might be failing

## Root Cause Analysis

### Most Likely Causes (in order of probability)

#### 1. **Puppeteer HTML Rendering Issue** (70% probability)
- The HTML generated by `generateStatementHTML()` is not rendering properly in Puppeteer
- Possible reasons:
  - **Inline CSS not applying**: The HTML uses inline styles which might not work in headless browser
  - **Font rendering**: Text might be invisible due to font color or styling
  - **Viewport size**: Puppeteer might be using wrong viewport dimensions
  - **Async content**: If HTML has async elements, they might not load before PDF generation

#### 2. **Zoho API Data Format Issue** (15% probability)
- The statement data from Zoho API might have unexpected structure
- Invoice fields might be named differently than expected
- Data might be missing or malformed

#### 3. **PDF-lib vs Puppeteer Conflict** (10% probability)
- The code uses both Puppeteer (for HTML→PDF) and pdf-lib (for test endpoint)
- There might be a library conflict or version incompatibility

#### 4. **Headless Browser Sandbox Issue** (5% probability)
- The `--no-sandbox` flag might be causing rendering issues
- Chromium might not have required dependencies in the sandbox environment

## Recommended Solutions

### Solution 1: Debug Puppeteer Rendering (IMMEDIATE)
**Priority**: HIGH  
**Effort**: 2-3 hours

**Steps**:
1. Save the generated HTML to a file for inspection
2. Add screenshot capability to Puppeteer to see what's being rendered

### Solution 2: Use pdf-lib Instead of Puppeteer (ALTERNATIVE)
**Priority**: MEDIUM  
**Effort**: 4-6 hours

**Rationale**: pdf-lib is a pure JavaScript PDF library that doesn't require Puppeteer/Chromium
- More reliable in sandboxed environments
- Smaller dependency footprint
- Deterministic rendering
